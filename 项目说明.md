## 一、项目是什么

**graphrag-kg-agent** —— 面向个人文档集合（技术论文 / GitHub 文档 / 产品需求文档）的端到端 **GraphRAG** 系统：
文档解析 → 切块 → 向量化 → 实体关系抽取 → Neo4j 知识图谱 → 向量召回+图谱邻域扩展检索 → **带可追溯引用**的回答，配一个清晰专业、带「像素 Agent 房间」动效的前端工作台。定位：个人学习 + 简历展示 + 公开可复现。

**技术栈**：Python 3.11+（实际用 conda `myself` 环境，3.12）/ FastAPI / Pydantic · Neo4j 5.26 + Vector Index（Docker）· OpenAI-compatible LLM（不绑厂商）· PyMuPDF · React 19 + Vite + TypeScript + Cytoscape.js。

**仓库**：https://github.com/JS7B/graphrag-kg-agent （公开）

## 三、worktree 多窗口工作流（本项目的开发方式）

本项目用 **「大脑 + 工人」多窗口并行** 开发，靠 git worktree 隔离：

```
E:\Mine\
├── graphrag-kg-agent\            [main]          ← 大脑窗口：规划/评审/合并/维护 main
├── graphrag-kg-agent-backend\    [feat/backend]  ← 后端工人窗口
└── graphrag-kg-agent-frontend\   [feat/frontend] ← 前端工人窗口
```

**硬约定**（详见 CLAUDE.md / AGENTS.md「多窗口协作约定」）：
1. 每条工作线 = 一个独立 worktree + 一个固定分支 `feat/<板块>`，人工建、全程不变。
2. 工人窗口只在自己分支 commit，**绝不碰 main、不自行合并**。
3. 交接信号 = 工人本地 commit + 口头通知大脑分支名 → 大脑读 diff 评审。
4. **合并权统一在大脑窗口**，评审通过才 merge 回 main。
5. 工人开工前先 `git merge main` 同步（否则基线落后，合并时出「误删假象」）。
6. 共享资源错峰：Neo4j（同一容器同库）、端口（后端 8000 / 前端 5173）跨 worktree 共享，并行写图谱/起服务要错开；`.env` 不随 worktree 复制，需手动放。
7. 收工回收 worktree + 工人启动的后台进程（uvicorn/vite 残留会占目录与端口）。

**学习记录约定（DEVLOG）**：引入新工具/配置/框架或完成非显然工作流后，追加学习记录——板块的写进该板块目录 `DEVLOG.md`（`backend/DEVLOG.md`、`frontend/DEVLOG.md`），跨板块/基础设施写进 `docs/DEVLOG.md`。

## 四、换机器恢复步骤（关键）

> **好消息：所有工作已全部合入 `main` 分支**（两条 feat 分支领先 main 均为 0），所以新机器只 clone main 就拿到全部代码，**不丢任何东西**。worktree 是本地机制，远程没有 feat 分支，新机器重新建即可。

### 1. clone + 装环境

```bash
git clone https://github.com/JS7B/graphrag-kg-agent.git
cd graphrag-kg-agent
```

环境前提（新机器要装）：
- **Docker Desktop**（Windows 需 WSL2 后端）
- **Anaconda/conda** + 一个 Python 3.11+ 环境（本项目原用名为 `myself` 的环境）
- **Node.js 18+**

### 2. 配 .env（必须，否则 LLM 功能跑不通）

```bash
cp .env.example .env
```
**务必填真实值**（当前最大坑，见第五节）：`OPENAI_BASE_URL` / `OPENAI_API_KEY` / `CHAT_MODEL` / `EMBEDDING_MODEL` / `EMBEDDING_DIM`（与 embedding 模型维度一致）/ `NEO4J_PASSWORD`。

### 3. 装依赖

```bash
# 后端（在你的 conda 环境，如 myself）
cd backend && pip install -r requirements.txt && cd ..
# 前端
cd frontend && npm install && cd ..
```

### 4. 启动（详见 运行说明.md）

```bash
docker compose up -d neo4j                                  # 1. Neo4j
cd backend && python -m uvicorn app.main:app --reload --port 8000   # 2. 后端
cd frontend && npm run dev                                  # 3. 前端 → localhost:5173
```

### 5.（可选）重建 worktree 继续多窗口开发

```bash
git worktree add ../graphrag-kg-agent-backend  -b feat/backend  origin/main
git worktree add ../graphrag-kg-agent-frontend -b feat/frontend origin/main
# 注意：每个新 worktree 要手动 cp .env 进去（前端还要各自 npm install）
```

## 五、已知问题与遗留事项

### ⚠️ 必须先处理（否则功能跑不通）
- **`.env` 的 LLM 配置**：换机器后 `CHAT_MODEL`/`EMBEDDING_MODEL` 等若仍是占位值 `your-xxx`，`/health/deps` 会显示 `llm: not_configured`，**问答、实体抽取、文档完整入库全部失败**（上传会卡在 embedding 报 Connection error）。CHAT_MODEL 需支持 JSON 输出模式（`response_format json_object`）。

### 📋 待办板块
- **公开展示与简历材料**（唯一未动板块）：完善 README、绘架构图、演示脚本、录 GIF、整理简历 bullet。

### 🔧 待优化（非阻塞，质量项）
- **实体召回率 57.1% < 70% 目标**（评估实测）：LLM 抽取漏了部分技术名词（React/Vite/TypeScript 等）。可调抽取 prompt 或加近义合并。其余指标达标（关系可用 100%、引用命中 100%、幻觉 16.7%）。
- **AgentRoom 像素瑕疵**：deleting 等状态道具会遮挡小人主体，待微调位置。
- **RunEvent 字段命名不统一**：`runId`/`createdAt` 走 camelCase，唯独 `timestamp_ms` 是 snake_case（前端已适配，能用，但风格不一致）。
- **rerank**：RERANK_MODEL 不可用时问答自动降级为按向量分排序，不影响主链路。

### ✅ 已验证可跑通
- 完整启动链路（Neo4j→后端→前端）2026-06-25 实测通过。
- 不依赖 LLM 的部分：健康检查、图谱/文档/chunk API、统一错误响应、失败处理（失败在写库前，不留脏数据）均正常。
- 后端全量测试 119 passed + 1 skip（真实 LLM 测试因无 key 而 skip）。
