# 后端说明

个人知识图谱 GraphRAG Agent 的后端工程实现说明。重点描述 **Agent 基础设施**——编排、上下文记忆管理、function calling——以及支撑它们的数据流与状态机。

## 1. 定位与技术栈

面向个人文档集合（技术论文 / GitHub 仓库文档 / PRD）的端到端 GraphRAG 系统。后端职责：文档解析与切块 → 向量化与图谱构建 → 实体关系抽取 → **Agentic RAG 问答**（ReAct 检索-反思循环）→ 对外提供 HTTP + SSE 接口。

| 维度 | 选型 |
|---|---|
| 语言 / 框架 | Python 3.12 / FastAPI / Pydantic |
| LLM | OpenAI-compatible 接口（chat / embed / rerank），配置全走 `.env` |
| 图谱与向量 | Neo4j 5 + Vector Index（`db.index.vector.queryNodes`），不另起向量库 |
| 解析 | PyMuPDF（PDF）+ 原生解析器（md / txt / 仓库目录） |
| 实时通信 | SSE（`sse-starlette`）推送 Run 进度事件 |
| Agent 编排 | **纯自研轻量 while 循环 + 原生 function calling，不引 LangGraph / LangChain** |

> 依赖见 `requirements.txt`，无版本锁定（conda `myself` 环境复现）。**没有引入任何 LLM 编排框架**——这是有意决策（见 §8）。

## 2. 工程结构

```
backend/
├── app/
│   ├── main.py            # FastAPI 工厂 + lifespan（建 Neo4j 连接 / RunStore / schema）
│   ├── config.py          # pydantic-settings，从仓库根 .env 读配置
│   ├── errors.py          # 统一错误响应 + 全局异常处理器
│   │
│   ├── clients/           # 外部服务薄封装
│   │   ├── llm.py         #   chat / chat_with_tools / embed / rerank / is_configured
│   │   └── graph.py       #   Neo4j 驱动生命周期
│   │
│   ├── parsing/           # 入库链路①：解析 + 分块（md/txt/pdf/仓库目录）
│   ├── extraction/        # 入库链路②：LLM 抽实体关系 → 归并 → 写图谱
│   ├── graph/             # 入库链路③：schema / embedding / writer / 向量检索
│   │
│   ├── qa/                # 问答链路（B 板块）
│   │   ├── agent.py       #   ★ Agentic RAG 核心：ReAct 循环 + function calling
│   │   ├── pipeline.py    #   线性 RAG（agent 降级时回退到它）
│   │   ├── context.py     #   上下文拼装 + Citation 闭环
│   │   ├── expand.py      #   实体关系邻域扩展
│   │   ├── rerank.py      #   重排（失败静默降级为截断）
│   │   └── prompt.py      #   问答 system prompt
│   │
│   ├── routers/           # HTTP / SSE 接口层（health / chat / documents / runs / graph）
│   └── runs/              # Run 进度追踪（models / store / tasks）
│
└── tests/                 # conftest 顶层共享 + 各模块测试
```

两条核心链路：**入库**（parsing → graph → extraction，写图谱）与**问答**（qa，查图谱 + 生成）。

## 3. 三层数据流

所有问答请求走「协议 → 编排 → 执行」三层，职责严格分离：

```
HTTP 请求                 编排层                      执行层
─────────               ──────────                  ──────────
POST /api/chat   →   routers/chat.py           →   runs/tasks.py::run_chat
  {question}         建 Run，立即返 runId          (BackgroundTask，异步)
                         │
                    runs/store.py (RunStore)        │ on_event 回调
                    内存队列 + SSE fan-out  ←────────┘
                                                      │
                                                 qa/agent.py
                                                 answer_question_agentic
                                                 (ReAct 循环)
```

- **协议层**（routers）：把 HTTP 请求转成 Run，立即返回 `runId`，不阻塞。SSE 端点 `GET /api/runs/{id}/events/stream` 让前端订阅进度。
- **编排层**（runs/tasks）：串接业务步骤，全程 try/except 吞异常并 emit error 终态（否则前端 SSE 流永不关闭、像素 Agent 卡住）。
- **执行层**（qa/agent）：纯检索逻辑，不依赖 RunStore，通过 `on_event` 回调把内部决策通知上层，保持可单测。

## 4. Agent 基础设施（核心）

这是后端的工程重心。下面分四部分讲：**编排循环**、**上下文与记忆管理**、**function calling 协议**、**引用可追溯闭环**。

### 4.1 编排：ReAct 检索-反思循环

核心函数 `qa/agent.py::answer_question_agentic`，默认参数 `max_turns=4 / top_k=10 / rerank_top_n=5`。

```
answer_question_agentic(question)
  │
  ├─ messages = [system(ReAct 指令), user(question)]   ← 单次会话局部变量
  ├─ evidence_pool = {}   (dict[chunk_id → ChunkHit]，跨轮去重累积)
  ├─ paths_acc = []       (关系路径，按四元组去重累积)
  │
  └─ for turn in range(max_turns):          ← for 循环 + 硬上限防死循环
       │
       ├─ msg = llm.chat_with_tools(messages, tools, tool_choice="auto")
       │       ↑ 端点不支持 tools → BadRequestError → 抛 ToolCallingUnsupported
       │
       ├─ messages.append(assistant_msg)     ← 必须保留 tool_calls 字段
       │
       ├─ if 无 tool_calls:                  ← 路径①：模型认为证据足够
       │     emit(CHECKING, "评估证据充分性")
       │     return _generate_final_answer(...)
       │
       └─ for tc in tool_calls:              ← 一轮可并行返回多个，全执行全回传
             dispatch(tc) → tool_result       ← 执行检索，累积证据池
             emit(stage_by_tool(tc.name))     ← 按工具类型映射事件
             messages.append({role:tool, tool_call_id, content:tool_result})

     # 路径②：达 max_turns 硬上限
     return _generate_final_answer(...)       ← 用已有证据强制生成兜底答案
```

**两条终止路径都走同一个 `_generate_final_answer`**，保证引用闭环一致。关键设计：

- **`on_event` 解耦**：agent 不直接依赖 RunStore，靠注入的回调 `Callable[[Stage, str], None]` 把"现在在干嘛"通知上层。`run_chat` 注入的回调把 stage 映射成 SSE 事件，驱动前端像素 Agent 动画。
- **证据池跨轮累积**：`vector_search` 每轮召回的 chunk 进 `evidence_pool`（dict 键去重，保留完整 `ChunkHit` 作来源凭证），最终统一进引用闭环。回传给模型的文本只含本轮结果（裁剪到 200 字控 token），不是整个池。
- **工具失败不崩**：单个工具执行失败（如某次查询出错）不中断循环，而是把错误字符串作为 tool result 回传，让模型自行决策（符合 ReAct 哲学）。

### 4.2 上下文与记忆管理

**结论先行：当前是单轮问答，无跨请求持久化记忆。** Agent 的"多轮"指单次请求内的 ReAct 循环，不是用户对话历史。

#### 上下文拼装（`qa/context.py::build_context`）

把检索到的 chunks + 关系路径拼成给 LLM 的文本，并同步生成 Citation：

```
【文档片段】
[1] chunk 原文...
[2] chunk 原文...
[3] chunk 原文...

【相关实体关系】
React -[uses]-> Vite
Vite -[integrates_with]-> TypeScript
```

关键：`enumerate(chunks, start=1)` 让正文里的 `[n]` 角标与 `Citation(index=n)` **同步生成、共用同一个序号**，这是引用可追溯的锚点。每条 Citation 携带 `chunk_id / document_id / location / snippet(前120字)`。

#### 上下文量控制

- 召回上限：`top_k=10`（向量召回）
- 重排截断：`rerank_top_n=5`（重排后只留 5 条进上下文）
- 工具结果裁剪：`_SNIPPET_LEN=200`（回传给模型的每条 chunk 正文前 200 字）

#### 记忆的边界（诚实说明）

| 有 | 没有 |
|---|---|
| 单次请求内的 ReAct 对话历史（messages 局部变量） | 跨请求的 session / conversation / memory |
| Agent 多轮检索的证据累积（evidence_pool） | 用户多轮提问的记忆连贯 |
| RunEvent 进度事件流（进程内，重启即丢） | 对话历史的持久化存储 |

`POST /api/chat` 的请求体只有 `question: str` 一个字段，每次新建 Run，不关联历史。这是当前阶段的取舍（见 §8）——单轮 + 精准检索已满足个人项目需求。

### 4.3 function calling 协议

LLM 客户端（`clients/llm.py`）提供两个对等函数：

| 函数 | 签名 | 返回 | 用途 |
|---|---|---|---|
| `chat` | `(messages, *, response_format=None)` | `str`（首条回复文本） | 线性问答、实体抽取 |
| `chat_with_tools` | `(messages, *, tools, tool_choice="auto")` | `ChatCompletionMessage`（含 tool_calls） | Agent 循环决策 |

两者都从 `settings.chat_model` 取模型名，从 `OpenAI(base_url, api_key)` 建客户端。配置全走 `.env`，绝不硬编码。

#### 工具集（`agent.py::_TOOLS`，OpenAI function calling JSON Schema）

两个工具，`additionalProperties: false` 防幻觉参数：

| 工具 | 参数 | 内部实现 | 何时用 |
|---|---|---|---|
| `vector_search` | `query: str` | embed → `search_chunks`(向量召回) → `rerank_chunks` | 查具体事实 / 技术细节 / 文档内容 |
| `expand_entity` | `chunk_ids: list[str]` | `expand_entities`(Cypher 遍历邻域) | 已有片段，想挖实体间关联 |

模型在每轮决策：调哪个工具、传什么参数、何时停止。`tool_choice="auto"` 让模型自主。

#### 降级机制

```
chat_with_tools 端点不支持 tools
  → 抛 openai.BadRequestError
  → agent 转成 ToolCallingUnsupported(RuntimeError)
  → run_chat 捕获
  → 回退线性 answer_question（pipeline.py）
  → emit 固定三事件序列（SEARCHING→CHECKING→WRITING）
```

**区分两种失败**：「端点能力不支持」触发整体降级；「单次工具调用失败」不降级，错误回传给模型继续决策。`is_configured()` 判定 LLM 是否配置（base_url/api_key 非空且不含占位标记）。

### 4.4 引用可追溯闭环

项目的硬要求：每个回答必须能追溯到原始 chunk。三段闭环：

```
检索 chunks ──build_context 编号──> 正文角标 [n]
    │                                   │
    │ Citation(index=n, chunk_id, ...)   │ LLM 在答案正文里引用 [n]
    │                                   │
    └───────────────────────────────────┘
                    ↓
        GET /api/chunks/{chunk_id} → 原文
```

`_generate_final_answer` 还做了**输出净化**：正则 `\[(\d+)\]` 提取答案正文真实出现的角标号，只保留对应的 Citation——未被引用的 chunk 不进结果，防止 LLM 编造引用。

## 5. 入库链路（简述）

与问答并行，写入图谱：

```
POST /api/documents (文件)
  → runs/tasks.py::run_ingest (BackgroundTask)
      ├─ parsing/parse_file      → ParsedDocument + Chunks（MAX_CHARS=800, OVERLAP=150）
      ├─ graph/embed_chunks      → 每块 embedding（settings.embedding_dim，默认 3072）
      ├─ graph/ingest_document   → Document + Chunk 节点 + 向量入库
      └─ extraction/extract_and_ingest
           ├─ 逐 chunk LLM 抽实体关系（JSON 模式，max_attempts=3 指数退避）
           ├─ merge 归并（同实体多称谓合并）
           └─ writer 写图谱（:Entity / :RELATES，关系类型作属性）
```

Stage 序列固定线性：UPLOADING→PARSING→EXTRACTING→INDEXING→IDLE。每个 chunk 保留 `char_start/char_end/page/heading_path`，是引用落地的位置凭证。

## 6. RunEvent 状态机与 SSE

### Stage 枚举（前端契约锁定，12 个）

`IDLE / UPLOADING / PARSING / EXTRACTING / LINKING / INDEXING / SEARCHING / CHECKING / WRITING / DELETING / REBUILDING / ERROR`。值即前端消费的字符串，不得改名。

Agent 运行时，stage 序列由工具调用**动态决定**（vector_search→SEARCHING、expand_entity→LINKING、生成→WRITING），多轮问答会反复 SEARCHING/CHECKING，前端像素房间真正跟着 Agent 多轮决策走。

### RunStore（进程内，不持久化）

- `dict[run_id → Run]` 存 Run + events 历史；`dict[run_id → list[Queue]]` 存订阅者。
- `append_event`：存历史 + 更新 Run 终态 + 向所有订阅队列 fan-out。
- `subscribe`（async）：新建 Queue，**先投递历史事件防漏**，再注册订阅。

重启即丢——Run 是瞬态进度信号，前端刷新会重新查 Document 状态（落 Neo4j）兜底。

## 7. 配置与启动

配置走仓库根 `.env`（非 `backend/.env`），`config.py` 用 `pydantic-settings` 读取，`@lru_cache` 单例。关键字段：`OPENAI_BASE_URL / OPENAI_API_KEY / CHAT_MODEL / EMBEDDING_MODEL / EMBEDDING_DIM(默认3072) / NEO4J_* / MAX_UPLOAD_MB(默认10)`。

启动（详见仓库根 `运行说明.md`）：

```bash
docker compose up -d neo4j                          # Neo4j（7474/7687）
cd backend
python -m uvicorn app.main:app --reload --port 8000 # 后端（8000）
```

lifespan 启动时 `ensure_schema` 幂等建约束 + 向量索引（失败仅告警不阻断）。健康检查：`GET /health`、`GET /health/deps`。

## 8. 关键设计取舍

| 决策 | 理由 |
|---|---|
| 自研循环，不引 LangGraph/LangChain | 单 agent + 内部工具场景，手写 while 循环代码量小、零新依赖、编排 100% 自控；框架价值在多 agent 编排/状态机/checkpointer，本项目用不上 |
| 单轮问答，无对话记忆 | 单轮 + 精准检索已满足个人项目需求；记忆引入 session/存储/上下文窗口管理复杂度，属超前设计 |
| 工具只做两个（搜索 + 扩展） | 决策空间小、可控、token 省；额外工具（如独立 rerank、图谱路径查询）按需再加，避免过早堆砌 |
| 降级只在端点不支持 tool calling 时 | 区分「能力不支持」（整体降级）与「单次工具失败」（回传错误让模型决策），不掩盖偶发问题 |
| RunStore 内存不持久化 | Run 是瞬态进度信号，Document 落 Neo4j 兜底；持久化引入存储复杂度，当前不值得 |
| 事件按工具类型动态映射 | 满足「动画状态来自真实 RunEvent」红线，多轮问答的真实节奏反映在前端 |
