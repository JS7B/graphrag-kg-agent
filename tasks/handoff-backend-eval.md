## 二、硬指标（评估要验证的目标，来自 AGENTS.md）

- 样本文档解析成功率 **100%**，每个 chunk 保留来源与位置
- 实体召回率 **≥ 70%**，关系可用率 **≥ 60%**
- 引用命中率 **≥ 70%**，明显幻觉率 **≤ 20%**

## 四、工作内容（对应 todo「评估与质量保障」13 项）

1. **准备样本文档集**（放 `samples/`，注意版权/隐私——用可公开的：如 arXiv 论文、开源项目 README、自造的需求文档）。覆盖三类:技术论文(PDF)、GitHub 文档(md)、产品需求文档。数量够评估即可（如 3-5 篇）。
2. **准备问题与参考答案**：针对样本文档设计若干问题 + 人工参考答案（放 `evals/`，如 `questions.jsonl`）。
3. **标注关键实体和关系**：人工标注每篇样本的关键实体/关系作为 ground truth（评估召回率用）。
4. **编写 eval 脚本**（`evals/run_eval.py` 之类）：
   - 跑解析 → 统计解析成功率、chunk 来源完整性
   - 跑抽取 → 对比标注，算实体召回率、关系可用率
   - 跑 GraphRAG 问答 → 算引用命中率（引用的 chunk 是否真支撑答案）、明显幻觉率（答案是否含无依据内容）
5. **统计四项指标**并输出报告。
6. **补测试**：补后端单元测试薄弱处 + 核心流程集成测试。

## 五、边界与约定

- **指标定义要写清**：召回率/命中率/幻觉率怎么算、分母分子是什么，写进脚本注释或 `docs/evaluation.md`。这是可复现的关键。
- **样本零提交风险**：可公开样本可入 `samples/`；有版权风险的放 `samples/private/`（已被 .gitignore）。
- **真实 LLM + Neo4j**：评估要调真实 LLM、读写 Neo4j。**Neo4j 是跨 worktree 共享资源**，前端窗口此时也可能在打后端/连库——**错峰使用**，评估写图谱用独立 document_id（如 `eval_` 前缀）便于清理，别污染。
- **简单优先**：评估脚本不必做成框架，能跑出指标、可复现即可。指标算法允许人工辅助判断（如幻觉率可半自动+人工核），但要说明方法。
- **装新依赖先问**（如需要 jsonlines、评估库等），遵守全局约定。
- **DEVLOG**：完成后在 `backend/DEVLOG.md` 追加学习记录（评估方法、指标定义、踩坑）。
- **产出 `docs/evaluation.md`**：评估集构成、指标定义、复现步骤（AGENTS.md 导航里标了"待建"）。
