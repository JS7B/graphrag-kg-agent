# Lessons（经验沉淀）

记录纠错后的模式，每次会话开始先复习。

---

## L1：先认准"价值/技能落在哪"，再谈做什么

**场景**：把 Agent Cartographer 从企业平台蓝图（星枢）降配为个人项目时，我反复在"harness 是奖杯还是底座""KG 该自建还是套框架"上摆动，被用户连续纠正三次。

**根因**：没先锁定"这个项目的技能/价值落点"，就去争论各部分的取舍，导致每次都在错误的锚点上优化。

**正确次序**：
1. 先定**最终目标**（为什么做）——本项目 = 产出好用的 KG onboarding 产物。
2. 再定**重点投入/技能落点**——本项目 = harness（自建运行编排运行时）。
3. 二者关系 = **手段服务目标**：每处 harness 设计都要能追溯到"让产出更可靠/更省钱/更可审计"。

**如何应用**：接到"把 X 改造/降配/落地"类任务，先问一句"这个项目的价值和技能落点是什么"，锁定后再讨论各模块取舍。别在没锚点时争论细节。

---

## L2：从重模板派生轻产品时，治理仪式会"搭便车"跟来

**场景**：Cartographer 蓝图正确砍掉了星枢的"规模"（多租户/SaaS/配额），却几乎原样继承了"治理机械"（多角色/审批/多状态机）——在个人单人语境里是"借来的企业排场"。

**根因**：治理对象（角色、门、状态机）是编织进父文档每一章的，降配范围时它们会隐性跟随。

**如何应用**：派生/降配任何重模板时，单独做一遍"治理减法"——逐个问"这个治理对象在新语境（单人/本地/学习）下还成立吗"。本项目的减法清单见设计文档 §3.3。判据：广度让作品集项目**更像没做完的克隆**，深度才是信号——**深度 > 广度**。

---

## L3：分清"我靠造它学到的"与"agent 对外产出的"

**场景**：我一度把"上下文工程可视化"当成产品主角，被用户点出：上下文怎么装配是**我构建时要理解的 harness 内部逻辑**，不一定要做成 agent 产出的花哨功能。

**根因**：把"开发者的学习底座"和"用户拿到的产物"揉成一团，催生过度工程（高投入、低产出价值的功能）。

**如何应用**：设计 agent 类项目时三分——①我靠造它学到的（内部逻辑，吃透+讲得清即可）②agent 对外产出的（真正的产品功能）③产物里正常可查的观测（该有、好做、够用，别堆仪表盘）。别把①硬塞成过度工程的产品功能。

---

## L4：优化派生文档 = "重新选主角"，不是"加内容"

**场景**：原蓝图的病不是写错，而是把真正的主角（失败模式清单 + 证据/置信分层）埋进 17 章平铺里，与企业残留并列。

**如何应用**：优化一份"什么都对但没重点"的文档时，先判断"主角被埋了吗"，优先做"提主角、砍配角、点破已隐含的好设计"，而非堆新内容。本项目把"玻璃盒可追溯产物"提为护城河，把失败模式清单（蓝图§15）保留强化。

---

## L5：研究外包给并行子代理，保持主上下文干净

**场景**：harness 骨架选型需要调研 Run 生命周期 / 编排拓扑 / KG 管线 / 上下文与可观测四条线。分发 4 个并行子代理各盯一个方向，主上下文只接收压缩结论。

**如何应用**：复杂选型/调研，按"一个子代理一个方向"分发并行，要求"final message 即结构化结论，不堆原始材料"，并要求标注"未能验证"项。对齐 CLAUDE.md 子代理策略。注意：子代理环境可能 WebFetch 受限，API 签名类结论需在实现阶段现场核对。
